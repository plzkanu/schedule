## 1. 사전 준비

### 1-1. 필요 도구

- **Node.js** 20.x 권장
- **npm** (Node.js에 포함)
- **Git**
- [Supabase](https://supabase.com/dashboard) 프로젝트 (PostgreSQL)
- (배포 시) [Replit](https://replit.com) 계정

### 1-2. Supabase DB 마이그레이션 (최초 1회)

앱이 사용하는 테이블은 `it_` 접두사를 씁니다. 입찰 시스템(`khnp_bid_*`)과 **같은 Supabase 프로젝트를 공유해도** 테이블명이 겹치지 않습니다.

1. Supabase Dashboard → **SQL Editor**
2. 저장소의 `supabase/apply-all-migrations.sql` 전체를 붙여넣기 → **Run**
3. **Table Editor**에서 주요 테이블 생성 확인  
   `it_projects`, `it_tasks`, `it_users`, `it_tech_capabilities`, `it_reviews` 등
4. 필요 시 **Project Settings → API → Reload schema** (또는 1~2분 대기)

개별 파일로 적용하려면 `supabase/migrations/` 아래 SQL을 번호 순으로 실행합니다.

| 파일 | 설명 |
|------|------|
| `001_it_projects.sql` | 프로젝트 |
| `002_it_tasks.sql` | 태스크 |
| `003_it_task_dependencies.sql` | 태스크 의존 관계 |
| `004_it_activity_logs.sql` | 활동 이력 |
| `005_it_users.sql` | 사용자·로그인 |
| `006` ~ `011` | RLS 해제, 컬럼 추가, 기술 확보 등 |
| `012_it_project_issues.sql` | 프로젝트 이슈 |
| `013_it_reviews.sql` | 검토 |

### 1-3. RLS 오류 예방 (권장)

등록·수정 시 `new row violates row-level security policy` 오류가 나면 SQL Editor에서 실행:

```sql
ALTER TABLE it_projects DISABLE ROW LEVEL SECURITY;
ALTER TABLE it_tasks DISABLE ROW LEVEL SECURITY;
ALTER TABLE it_task_dependencies DISABLE ROW LEVEL SECURITY;
ALTER TABLE it_activity_logs DISABLE ROW LEVEL SECURITY;
ALTER TABLE it_users DISABLE ROW LEVEL SECURITY;
NOTIFY pgrst, 'reload schema';
```

또는 `supabase/migrations/007_it_tables_disable_rls.sql` 내용을 실행합니다.

## 2. 로컬 실행

### 2-1. 소스 받기

```bash
git clone https://github.com/plzkanu/schedule.git
cd schedule
```

이미 클론한 폴더가 있다면:

```bash
cd schedule
git pull origin main
```

> 기본 브랜치가 `master`이면 `main` 대신 `master`를 사용하세요.

### 2-2. 패키지 설치

```bash
npm install
```

### 2-3. 환경 변수 설정

`.env.local.example`을 복사해 `.env.local`을 만듭니다.

**Windows (PowerShell)**

```powershell
Copy-Item .env.local.example .env.local
```

**macOS / Linux**

```bash
cp .env.local.example .env.local
```

`.env.local` 예시:

```env
NEXT_PUBLIC_SUPABASE_URL=https://xxxxxxxx.supabase.co
SUPABASE_SERVICE_ROLE_KEY=your-service-role-key
AUTH_SECRET=your-random-secret

# 회사 VPN/방화벽에서 Supabase fetch failed 시에만 (개발·내부망 한정)
# SUPABASE_SSL_VERIFY=0
```

| 변수 | 설명 |
|------|------|
| `NEXT_PUBLIC_SUPABASE_URL` | Supabase 프로젝트 URL |
| `SUPABASE_SERVICE_ROLE_KEY` | Service Role Key (서버 전용, **절대 Git에 커밋 금지**) |
| `AUTH_SECRET` | 세션 쿠키 서명용 랜덤 문자열 (`openssl rand -base64 32` 등) |
| `SUPABASE_SSL_VERIFY` | TLS 오류 시 `0` (운영 환경에서는 가급적 사용하지 않음) |

`AUTH_SECRET`은 아래처럼 생성할 수 있습니다.

```bash
# macOS / Linux / Git Bash
openssl rand -base64 32
```

### 2-4. 개발 서버 실행

```bash
npm run dev
```

브라우저에서 접속:

**http://localhost:3001**

포트는 `package.json`의 `dev` 스크립트에서 `-p 3001`로 지정되어 있습니다.

### 2-5. 로그인 테스트 계정

마이그레이션 시드(`005_it_users.sql`) 기준:

| 역할 | 아이디 | 비밀번호 | 권한 |
|------|--------|----------|------|
| 관리자 | `admin` | `admin123` | 전체 + 사용자 관리 |
| 관리자 | `hskim` | (입찰 시스템과 동일) | 관리자 · IT팀 |
| IT팀원 | `member1` | `member123` | 프로젝트/태스크 CRUD |
| 조회 | `viewer1` | `viewer123` | 조회 전용 |

아이디 또는 이메일(`hskim@soosan.com` 등) 모두 로그인에 사용할 수 있습니다.

### 2-6. 프로덕션 빌드 로컬 확인 (선택)

배포 전 동작을 로컬에서 검증할 때:

```bash
npm run build
npm run start
```

`start`도 포트 **3001**을 사용합니다.

## 5. Replit 배포

### 전체 흐름

```
로컬 수정 → GitHub push → Replit Shell 동기화 → build → Republish
```

### 5-1. Replit 프로젝트 생성 (최초 1회)

1. [replit.com](https://replit.com) 로그인
2. **Create Repl** → **Import from GitHub**
3. `plzkanu/schedule` 선택
4. 템플릿: **Node.js**

### 5-2. Replit 설정 파일 (권장)

프로젝트 루트에 `.replit` 파일을 추가합니다. (Replit에서 **Show hidden files** 활성화)

```toml
run = "npm run dev"
entrypoint = "package.json"
hidden = [".config", ".next", "node_modules"]
modules = ["nodejs-22"]

[nix]
channel = "stable-24_11"

[deployment]
build = ["npm", "run", "build"]
run = ["npm", "run", "start"]
deploymentTarget = "cloudrun"

[[ports]]
localPort = 3000
externalPort = 80
```

Replit 배포 시 컨테이너 외부에서 접속하려면 `package.json`의 `start`를 아래처럼 맞추는 것이 좋습니다.

```json
"start": "next start --port 3000 --hostname 0.0.0.0"
```

로컬 개발은 기존처럼 `dev`에서 3001 포트를 써도 됩니다.

### 5-3. Replit Secrets (필수)

**Workspace Secrets**와 **Deployments → Secrets** 모두 확인하세요.  
**배포용 Secrets가 없으면** 프로덕션에서 Supabase 연결이 실패합니다.

| Secret | 설명 |
|--------|------|
| `NEXT_PUBLIC_SUPABASE_URL` | Supabase URL |
| `SUPABASE_SERVICE_ROLE_KEY` | Service Role Key |
| `AUTH_SECRET` | 세션 서명 시크릿 (로컬과 동일하게 새로 생성 가능) |
| `SUPABASE_SSL_VERIFY` | (선택) 내부망 TLS 오류 시 `0` |

### 5-4. 배포할 때마다 (Replit Shell)

```bash
git fetch origin main && git reset --hard origin/main && npm install && npm run build
```

브랜치가 `master`이면 `main`을 `master`로 바꿉니다.

### 5-5. Republish

1. Replit 상단 **Deploy** → **Republish** (또는 **Deploy**)
2. 배포 타입: **Autoscale** (일반 웹앱 권장)
3. 완료 후 `https://your-app.replit.app` 접속

## 7. 문제 해결

### `Could not find the table 'public.it_projects'`

- Supabase SQL Editor에서 `supabase/apply-all-migrations.sql` 실행
- `.env` / Secrets의 URL이 **테이블을 만든 프로젝트**와 같은지 확인
- 스키마 리로드 후 개발 서버·Republish 재시도

### `new row violates row-level security policy`

- [1-3. RLS 오류 예방](#1-3-rls-오류-예방-권장) SQL 실행

### Supabase `fetch failed` (회사망/VPN)

- `.env.local` 또는 Replit Secrets에 `SUPABASE_SSL_VERIFY=0` 추가 (내부망 한정)
- 개발 서버 / Republish 재시작

### Replit `an open port was not detected`

- `package.json` `start`에 `--hostname 0.0.0.0` 추가
- 포트와 `.replit`의 `localPort` 일치 확인

### Replit에 최신 코드가 반영되지 않음

```bash
git log -1          # 최신 커밋인지 확인
git fetch origin main && git reset --hard origin/main
npm install && npm run build
```

이후 **Republish** 다시 실행

### 로그인은 되는데 데이터가 안 보임

- Secrets 3종(`NEXT_PUBLIC_SUPABASE_URL`, `SUPABASE_SERVICE_ROLE_KEY`, `AUTH_SECRET`) 확인
- Supabase Table Editor에서 `it_*` 테이블·데이터 존재 여부 확인

### 역할 변경 후 권한이 안 바뀜

- 역할을 DB에서 바꾼 뒤에는 **로그아웃 후 다시 로그인**하세요.

### Replit에서 직접 수정 후 화면·기능이 망가졌을 때 (복구)

Replit 에디터·AI로 코드를 바꾸면 **GitHub와 다른 버전**이 되어 로컬과 화면이 달라지고, 빌드 오류·기능 장애가 납니다.  
**Replit에서는 코드를 수정하지 말고**, 아래 순서로 GitHub 기준으로 되돌립니다.

#### 1) 로컬에서 정상 코드를 GitHub에 올림 (Cursor)

```bash
git add .
git commit -m "fix: Replit 배포 설정 및 권한 모듈 분리"
git pull origin main --rebase
git push origin main
```

#### 2) Replit Shell에서 강제 동기화 (Replit 변경분 전부 폐기)

```bash
git fetch origin main
git reset --hard origin/main
git clean -fd
npm install
npm run build
```

- `git clean -fd`는 Replit에서만 생긴 추적되지 않은 파일을 삭제합니다.
- **주의:** Replit에서만 수정한 내용은 모두 사라집니다.

#### 3) Secrets 확인 후 Republish

**Tools → Secrets** 및 **Deployments → Secrets**에 아래 3개가 있는지 확인:

- `NEXT_PUBLIC_SUPABASE_URL`
- `SUPABASE_SERVICE_ROLE_KEY`
- `AUTH_SECRET`

**Deploy → Republish** 실행.

#### 4) 앞으로 지킬 규칙

| 하지 말 것 | 대신 |
|------------|------|
| Replit 에디터에서 UI/코드 수정 | Cursor에서 수정 → `git push` |
| Replit만의 변경을 GitHub에 섞기 | Replit은 `git reset --hard`로 받기만 |
| Workspace Secrets만 설정 | **Deployments Secrets**도 반드시 설정 |
